Anthropic 回滚 Fable 5 静默干预——48 小时内的政策反转

昨天（6/10）的雷达报道了 Fable 5 的静默干预机制，当时我说“影响了信任度”。结果不到 24 小时，Anthropic 就在 Wired 和 X 上正式认错：被标记的请求将回退到 Opus 4.8 并明确告知用户。这是罕见的大模型厂快速向社区让步的案例。

📍 背景

6/9 Anthropic 发布 Fable 5 时，系统卡披露了一个静默干预机制：当模型检测到用户试图开发竞争模型（如构建预训练流水线、ML 加速器设计）时，会通过提示修改、转向向量或 PEFT 悄悄地降低回答质量，而不回退到弱模型，也不会通知用户。这个设计被 Jeremy Howard、Simon Willison 等 KOL 猛烈批评，HN 上 #3 和 #21 帖子引发大量讨论。今天（6/11），Anthropic 通过 Wired 独家报道宣布改变：被标记的请求将回退到 Opus 4.8，且用户会看到回退提示。API 调用会返回明确的拒绝理由。

为什么这事重要

  • Anthropic 在 48 小时内让步 — 说明社区反弹的力量远超预期。如果你担心模型会被“暗改”，现在至少不会瞒着你了。
  • 透明度原则得到强化 — 之前“不可见”的 safeguard 现在变得可见。Anthropic 承认“不可见预防措施可以被探测，所以必须更健壮，这需要时间做好”。
  • 对开发者信任的修复 — 你使用 Fable 5 时，如果请求被拦截，会立刻知道原因，而不是困惑于模型变蠢了。
  • 间接证明“递归自改进”不再是科幻 — Anthropic 认为模型已经强大到能加速自身开发，所以他们必须防范恶意使用。这个论据本身就很值得关注。
🎯 今天就能做的事
  1. 读 Wired 报道和 Anthropic 的声明 — Wired 文章 和 @ClaudeDevs 的 X 帖子 了解完整细节。
  2. 重新评估是否使用 Fable 5 — 现在被拦截时你会知道，但仍然存在“哪些问题会被拦截”的不确定性。如果你做的是常规办公/代码工作，99.97% 的请求不受影响。
  3. 关注 API 回退机制的上线 — 服务器端回退将在未来几天部署。
  4. 继续试用 Fable 5 — 用 llm -m claude-fable-5 或 Claude Code，体验其真实能力，注意观察是否出现回退。

为什么 Anthropic 会这样做？ — 官方解释：不可见预防措施可以被探测和反向工程，所以他们优先部署了不可见版本以快速发布，但现在承认“做错了权衡”。新措施采用可见回退，和他们在网络安全、生物学、化学领域的干预一致。

这够了吗？ — Simon Willison 表示“比之前好多了”，但 Jeremy Howard 仍然不满，认为任何形式的限制都应该被移除：“如果你想减速，就不应该让自己也使用最强模型进行前沿 AI 研究。” 社区的分歧仍然存在。

对 clawii 的影响 — 你是补偿经理，日常使用 vibe coding，基本不会触发这些限制。但这次事件展示了：1）模型提供商可以静默操控输出；2）社区压力可以迫使改变。值得持续关注。

🔭 后续关注什么

1）API 的回退理由字段是否按时上线；2）是否有真实案例触发回退并被用户报告；3）其他厂商（OpenAI, Google）是否跟进类似政策或更加透明；4）Anthropic 是否会进一步缩小触发范围。

Google DiffusionGemma——开源多模态模型，500+ token/s

Google 去年 5 月短暂发布过一个实验性的 Diffuson 模型，之后没有声音了。今天他们以 Gemma 系列的形式回来了：DiffusionGemma 26B-A4B，Apache 2.0 开源，推理速度快到离谱（500+ token/s），NVIDIA 免费托管。

📍 背景

Google 在 2025 年 5 月悄悄放出了一个 "Gemini Diffusion" 预览，我当时试用时跑出 857 token/s 的速度。之后 Google 没有再提，大家都以为这个项目停了。结果昨天（6/10）他们以 Gemma 系列的新成员身份正式发布 DiffusionGemma 26B-A4B it 模型，Apache 2.0 许可，完全开源。模型是图像到文本的多模态模型（以前叫 any-to-any），但核心能力是图像理解 + 文本生成。NVIDIA 的 NIM 云 API 免费提供推理。

为什么这事重要

  • Apache 2.0 开源 — 你可以下载权重、本地运行、修改、商用。Google 通过 Gemma 系列真正践行了开源。
  • 速度快到离谱 — Simon Willison 用 time uv run generate.py 测试，生成 2409 token 只用了 4.4 秒，相当于 500+ token/s。在 NVIDIA NIM 上甚至更快。
  • 26B 总参数 / 4B 激活（MoE） — 和 Gemma 4 12B 类似的架构，但总参更大、激活参更小，适合在消费级显卡上运行（如果有量化版本）。
  • 推动多模态模型可及性 — 以前你需要 Gemini API 或 GPT-4V 才能做复杂的图像理解；现在有一个开源的、速度极快的替代品。
🎯 今天就能做的事
  1. 试用 NVIDIA NIM 免费 API — 用 NVIDIA NIM 直接上传图片提问。
  2. 下载权重 — HuggingFace 上 google/diffusiongemma-26B-A4B-it 已可用。
  3. 等待量化版本 — unsloth 已经发布了 GGUF 版本，你可以用 llama.cpp 在本地跑。

有什么不同 — 和 Gemma 4 12B 不同，DiffusionGemma 采用 MoE 架构，26B 总参数只有 4B 激活，因此推理效率极高。它走的是“扩散 + 自回归”混合路线，但使用上就是一个标准的文本输出模型。

跑在什么之上 — 模型权重已发布，支持 HuggingFace Transformers，NVIDIA NIM 提供免费推理。建议使用 unsloth 的 GGUF 版本本地运行。

上手难度 — 最简单：用 NVIDIA NIM 无需配置。中等：用 unsloth GGUF + llama.cpp 本地跑。

诚实说缺点 — 官方只发了一个 it 版本（指令微调），没有 base 版。模型仅支持图像输入（不是 any-to-any）。还没有看到独立的社区评测。

对 clawii 来说能做什么 — 如果你需要快速理解图表、截图、手写笔记，可以用本地开源的模型完成，无需调用付费 API。如果量化版本能在 8GB 显存的笔记本上跑，那将是巨大的生产力提升。

🔭 后续关注什么

1）量化模型（GGUF / QAT）的下载量和社区评测；2）Simon Willison 或其他 KOL 的深度使用体验；3）Google 是否会发布更大的版本或 base 版本。

MoE (Mixture of Experts)
混合专家模型——总参数很大，但每次只激活一小部分，从而实现又快又大的效果。
NIM (NVIDIA Inference Microservice)

GGUF
用来在本地 CPU/GPU 上高效运行大模型的格式。unsloth 提供了 GGUF 量化版本。
