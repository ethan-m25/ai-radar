Harness engineering: OpenAI's official guide on how to actually use Codex well

Stop obsessing over which model is strongest — the real gap is in how you build the loop around it. OpenAI just published their own harness methodology, and the HN discussion is on fire.

📍 Context

Over the past few months, the coding agent toolchain has exploded: Claude Code, Codex, Cursor, and more. But many people are discovering that swapping in a stronger model doesn't automatically make your agent better. The key differentiator is the "harness" — the workflow loop, tool calling, feedback mechanisms, and error recovery. Today OpenAI published a Harness engineering article on developers.openai.com, walking through how to build agent-first workflows with Codex. It hit 207 points and 131 comments on HN, with serious discussion from experienced agent builders.

Why this matters

  • First systematic treatment of harness from OpenAI: Everyone's been figuring this out in isolation. OpenAI just gave you a framework — prompt structure, tool registration, error recovery, long-conversation management. Directly applicable whether you use Codex or Claude Code.
  • Bigger leverage than model upgrades: The article compares "great model + mediocre harness" vs "decent model + well-designed harness" — the latter wins on task completion. Spending 20 minutes tuning your agent workflow today may beat waiting for the next model release.
  • Principles apply beyond coding: Even though Codex is a coding agent tool, the principles (clear goal decomposition, checkpoint validation, human-in-the-loop) apply to any AI workflow — including your daily long-task usage of Claude or ChatGPT.

  1. Read the original: Harness engineering: Leveraging Codex in an agent-first world
  2. Audit your current workflow: Do you have explicit checkpoints? Retry logic on errors? If you're running "one prompt to rule them all," try splitting into 3-4 verifiable steps.

Quant giant Jane Street: "I design with Claude more than Figma now"

Not a design agency's hot take — a quant trading firm engineer says Claude is deeply embedded in his daily UI workflow, including generating HTML prototypes from screenshots.

📍 Context

Jane Street is famous for OCaml and quantitative trading — deeply technical, engineering-driven culture. Engineer Kevin Liu posted on HN about his shift from "intellisense-off terminal debugging" to "using Claude for UI design." The key isn't that Claude can generate UI — it's how he uses it as a "design thinking tool" rather than a final-output generator. The post scored 175 points with 145 comments, one of the hottest AI usage case studies on HN today.

Why this matters

  • Quant firm engineers doing frontend: Jane Street isn't a design company, but its engineers are using Claude to generate usable UI prototypes — iteratively, for real internal tools. This signals AI design has crossed the "toy" threshold.
  • Keyword: iteration: His workflow is "screenshot → throw at Claude for changes → download HTML → tweak → repeat." Not one-shot generation, but a closed-loop design iteration. For you as a non-developer, this means you can use Claude to build simple UI prototypes without learning any design tool.
  • Existential pressure on Figma: If more engineers and non-designers discover they can get usable interfaces from text + screenshots, tools like Figma face a shrinking addressable market.
🎯 Action — what you can do today
  1. Read the original: Jane Street blog post
  2. Try it yourself: Open Claude, upload a UI screenshot you want to improve, say "Make this more modern and output a usable HTML file." You'll be surprised at the result.
  3. Browse the HN discussion (thread) — other companies share similar practices.
🔬 Deep dive — how this differs from previous "AI design" attempts

vs existing tools (Canva / Figma AI): Those tools require learning their interfaces, and the generated code is often not editable. Jane Street's approach uses raw HTML that works in any browser, and developers can further modify the code.

What it runs on: Claude (the article doesn't specify the version, but given the timeline it's likely Sonnet 4.6 or Opus 4.7). The user iterates through Claude's chat interface or API.

How hard to use: Very low. You just need to upload images and understand basic HTML output. You, clawii, already vibe-code with Claude Code and know HTML — this workflow takes 5 minutes to learn.

Honest caveat: Generated UI still needs manual tweaking, especially for complex interactions (form validation, state management). But for rapid prototyping, the efficiency gain is 5-10x real.

🔭 What to watch for next

If similar use cases appear in non-tech companies (marketing, operations teams), it means "AI replacing design tools" is entering the mainstream. Also watch whether Figma accelerates its AI features in response.

Her · हेर — a detective for your Claude Code sessions

A tool that lets you replay every step of a Claude Code session — when your agent does something wrong, you finally know why.

📍 Context

Claude Code's session logs are detailed, but debugging is still painful: you don't know why it ran a particular command or where it went wrong. Her is an open-source tool (published on the Hugging Face blog) that scans Claude Code logs and presents each step's thinking, action, and result in a readable timeline — like a detective replaying a case. Useful for debugging complex agent tasks.

Why know the name

  • Solves a real pain point: When Claude Code or Codex does something you don't understand, you normally dig through raw logs. Her organizes them into a "story timeline," dramatically reducing debugging time.
  • Community traction: The HF blog post has 70+ reactions (early stage, not yet KOL-level, but worth watching).

Not a must-read: But if you already use Claude Code for multi-step tasks and occasionally get confused by its behavior, spending 5 minutes on this project could save hours of future debugging.
