Inkling 975B — The strongest US open-weights model, Apache 2.0

Hey clawii — Thinking Machines Lab just dropped Inkling, a 975B-parameter (41B active) MoE model under Apache 2.0, with 1M context and multimodal support (text + image + audio). Mira Murati, Soumith Chintala, John Schulman, and Lilian Weng are all endorsing it on X. This is the first genuinely competitive open-weights model from the US — American open-source has been lagging behind China (Hy3, GLM-5.2), and Inkling finally closes the gap.

📍 Context · Open-weights models have been dominated by China: Tencent Hy3 295B, Zhipu GLM-5.2 753B, DeepSeek V4 series. In the US, Meta's Llama series has a big user base but its capability has been overtaken by Chinese models. Thinking Machines Lab was founded by former OpenAI CTO Mira Murati, with a team including Soumith Chintala (PyTorch creator), John Schulman (RLHF inventor), and Lilian Weng (former OpenAI safety lead). Inkling is their first formal model release — 975B total params but only 41B active (MoE), keeping inference costs manageable. Apache 2.0 means you can freely use, modify, and redistribute it commercially.

Why this matters

  • The US finally has a competitive open-weights model — Previous US open models (Llama 4, Gemma 4) trailed Chinese models by 10-20% on coding and agent tasks. Inkling approaches Hy3 and GLM-5.2 on multiple benchmarks, and even leads on some tasks. For clawii, this means you can use an Apache 2.0 model for commercial projects without worrying about licensing.
  • Mira Murati + Soumith + Schulman lineup — This isn't a random startup. These people built GPT-4, PyTorch, and RLHF. Their model deserves serious attention.
  • 1M context + multimodal — Supports text, image, and audio input, with a 1M token context window. For vibe coding, this means you can throw your entire codebase at it and let the model understand the full picture.

🔬 Deep dive — how is Inkling different from Hy3 / GLM-5.2?
AspectExplanation
Core idea975B total params, 41B active (MoE). Supports text, image, and audio input, text output. 1M token context. Apache 2.0 license.
vs Tencent Hy3 295BHy3 is 295B total / 21B active, 256K context. Inkling has more total params (975B vs 295B), more active params (41B vs 21B), and larger context (1M vs 256K). But Hy3 has 192 experts; Inkling's expert count is undisclosed. They're close on coding benchmarks; Inkling is stronger on multimodal (supports audio).
vs Zhipu GLM-5.2 753BGLM-5.2 is 753B total, active count undisclosed. Inkling has more total params (975B), but GLM-5.2 is stronger on Chinese-language tasks. Both are Apache 2.0. GLM-5.2 leads on Coding Arena, but Inkling is more comprehensive on multimodal.
Tech implementationMoE (Mixture of Experts) — only a subset of parameters (41B) is activated per inference, keeping inference costs manageable. Supports vLLM, SGLang, Modal, Baseten, Databricks, and other platforms.
Setup difficultyLow. You can call it via API directly (Tinker, Together, etc.), or self-host if you have a multi-GPU cluster. For vibe coding users, just call the API.
Honest caveatInkling still trails top Chinese models (Hy3, GLM-5.2) on some benchmarks. Also, 975B total params means full deployment needs a lot of GPU power — not something an individual can run. But API usage costs are manageable.

Honestly: Inkling's significance isn't whether it's the strongest model — it probably isn't. Its significance is that the US finally has an Apache 2.0 open-weights model built by top-tier AI researchers. That's good for the entire open-source ecosystem. For clawii, it means you have more options — you're no longer forced to rely only on Chinese models or closed APIs.

Jargon decoder · MoE (Mixture of Experts) = An architecture where the model has multiple "expert" sub-networks, and only a subset is activated per inference, keeping costs manageable while maintaining large model capacity. Apache 2.0 = A permissive open-source license that allows commercial use, modification, and redistribution with minimal restrictions. Active parameters = The number of parameters actually activated during inference — MoE models have far more total parameters than active ones.

Grok Build open source — 844k lines of Rust, local-first coding agent

Hey clawii — xAI open-sourced Grok Build's entire codebase yesterday: 844,530 lines of Rust under Apache 2.0. This came after Grok Build got hammered by the community for uploading user data to Google Cloud by default. xAI urgently fixed it and open-sourced to rebuild trust. Now you can set up a local-first coding agent where all data stays on your machine. For privacy-conscious clawii, this is worth a look.

📍 Context · Grok Build is xAI's coding agent CLI tool, similar to Claude Code and Codex. Last week it was caught uploading user directories to Google Cloud by default — including SSH keys, password manager databases, etc. The community erupted. xAI first disabled the upload feature, then deleted all uploaded data, and finally open-sourced the entire codebase. Simon Willison analyzed the code and found 844k lines of Rust with only about 3% being third-party code — the rest is xAI's own work. The codebase includes tool implementations ported from Codex and OpenCode, with proper attribution.

Why this matters

  • Local-first coding agent now has an open-source option — Claude Code and Codex are both closed-source. Grok Build being open means you can self-host, audit, and modify it. For clawii handling sensitive data, this is a real alternative.
  • 844k lines of Rust as a reference — This is one of the largest open-source coding agent codebases out there. If you want to understand how coding agents work internally (system prompts, tool calling, subagent management), this is a fantastic learning resource.
  • Trust-rebuilding case study — xAI's response (disable upload → delete data → open source) is worth watching. If the community accepts this approach, it could push other companies to open-source their coding agents too.

🔬 Deep dive — how is Grok Build different from Claude Code / Codex?
AspectExplanation
Core ideaA Rust-based coding agent CLI supporting code generation, editing, debugging, and deployment. System prompts and subagent prompts are in the codebase (`prompt.md` and `subagent_prompt.md`).
vs Claude Code / CodexClaude Code and Codex are closed-source — you can't see what they're doing. Grok Build is fully open, so you can audit every line. Also, Grok Build supports local-first operation (using your own inference), while Claude Code and Codex require cloud APIs.
Tech implementation844k lines of Rust, including a Mermaid diagram terminal renderer, file system tools, Git integration, and subagent management. Tool implementations are partially ported from Codex and OpenCode (with attribution and license notices).
Setup difficultyMedium. You need a Rust build environment, or wait for community binaries. For non-developers, using xAI's cloud version is simpler (but watch your privacy settings).
Honest caveatGrok Build's privacy issue is fixed, but the upload module is still in the codebase (just disabled). If you self-host, make sure you're not sending sensitive data anywhere. Also, Rust compilation takes a while (~30 minutes).

Honestly: Grok Build's open-sourcing is more of a trust-rebuilding move than a mature open-source project. The code quality is decent (per Simon Willison), but the community needs time to validate it. For clawii, the biggest value is learning how coding agents work internally, not immediately replacing Claude Code with it.
