MiMo Code open-sourced — Xiaomi's 1000 token/s model is here

When the Radar first mentioned MiMo on 6/9, it was a closed preview. Today, Xiaomi open-sourced the full weights of MiMo-V2.5-Pro-UltraSpeed under Apache 2.0, hitting #13 on HN with 504 points. This is the fastest open-source Chinese LLM yet, with inference speed exceeding 1000 token/s.

📍 Context

On 6/9 we reported on Xiaomi's MiMo-V2.5-Pro-UltraSpeed model, then an unannounced preview in limited testing. Now Xiaomi has open-sourced MiMo Code (likely a code-optimized variant), with weights on HuggingFace under Apache 2.0. The HN post scored 504 points with active discussion. The model uses MoE architecture: 1T total parameters, 60B active, inference speed >1000 token/s, and a 1M-token context window.

Why this matters

  • Open-source + fast + large context — Apache 2.0 means you can download, modify, and use commercially. 1000+ token/s means even complex tasks feel near-instant.
  • First open-source MoE to hit 1000 token/s — The previous fastest open model (Nemotron 3 Ultra) was ~400 token/s. MiMo pushes the bar to 1000+.
  • Relevance for non-developers — If you use Claude Code or Codex for office automation and vibe coding, MiMo can run locally — no API fees, no latency worries.
  • Native Chinese excellence — Xiaomi is a Chinese company; MiMo's Chinese language capability likely far exceeds Western models of similar size.
🎯 Action — what you can do today
  1. Download the weights — HuggingFace: XiaomiMiMo/MiMo-V2.5-Pro-UltraSpeed (~120GB, needs decent storage).
  2. Wait for quantized versions — unsloth will likely release a GGUF version soon. Watch unsloth's model page.
  3. Read the HN discussion — HN thread (#13) has 277 comments with real community feedback.
  4. Evaluate local deployment — If you have a GPU with 48GB+ VRAM (RTX 4090, A6000), a quantized version might run. 1000 token/s locally rivals cloud API speed.

What's different — MiMo uses a unique "DFlash" quantization technique that maintains high performance at FP4 precision. Its MoE activation ratio is extremely low (60B/1T = 6%), making inference very efficient.

What it runs on — Weights are in PyTorch format (likely compatible with huggingface transformers). Needs NVIDIA GPU and inference engines like vLLM / TensorRT. Xiaomi hasn't released official inference scripts yet, but the open-source community usually catches up fast.

How hard to use — High: currently only BF16/FP4 weights available; you'll need to manually quantize for consumer GPUs. Best to wait for unsloth's GGUF version. If you have high-end hardware (8×A100 or single H100), you can run the FP4 version directly.

Honest caveats — 1) Release is rushed, documentation may be incomplete; 2) Community benchmarks are scarce, real-world performance unverified; 3) Huge parameter count means high download and storage costs.

What clawii can do with it — If you have local GPU resources, MiMo can replace Claude API for code generation and office automation — zero API cost, faster speed. But quantization tools aren't ready yet; wait 1-2 weeks.

🔭 What to watch for next

1) Whether unsloth or llama.cpp releases a quantized version within a week; 2) Independent community benchmarks (especially Chinese coding and office tasks); 3) Whether Xiaomi releases smaller variants (e.g., MiMo-V2.5-Nano); 4) API pricing and third-party hosting support (Together, Groq, etc.).
