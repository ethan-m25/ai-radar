Better Models: Worse Tools — New Claude models degrade tool call format

Clawii — if you use Pi or any non-Claude-Code agent harness with Claude, you might find Opus 4.8 and Sonnet 5 are actually worse at tool calls than older models. Armin Ronacher documented this in detail. HN front page with 180 points.

📍 Context · While building the Pi tool, Armin discovered that Opus 4.8 and Sonnet 5 invent extra fields in custom edit tool calls that don't exist in the schema, causing the call to be rejected. Older models (Sonnet 4.5, Opus 4.5) are cleaner. His theory: Anthropic RL-trained newer models to be better at Claude Code's built-in search-and-replace tool, which "polluted" third-party custom tools. This sparked a big HN discussion — many independent agent framework devs are hitting the same wall.

Why this matters

  • Directly affects the agent tools you use — If you use Pi, Continue, or any Claude-based agent, newer models might be worse than older ones. This impacts your vibe-coding efficiency.
  • Reveals a structural problem — Model companies optimize for their own tools via RL, breaking compatibility with third-party frameworks. This isn't a bug — it's misaligned incentives.
  • You can act on this today — Temporarily downgrade to Sonnet 4.5 or Opus 4.5, or switch to Claude Code's built-in edit tools.
🎯 What you can do today
  1. If you use Pi or another non-Claude-Code framework, roll back to Sonnet 4.5 or Opus 4.5 and see if tool calls get cleaner.
  2. Read Armin's blog post Better Models: Worse Tools for the full details.
  3. If you maintain your own agent harness, consider supporting multiple edit tool formats per model, or implement auto-detection.

GPT-5.5 Codex reasoning-token clustering causing performance degradation

Clawii — a HN post points out that GPT-5.5 Codex's reasoning-token clustering optimization may be degrading coding agent output quality. 287 points, 114 comments. If you use Codex, this directly affects your workflow.

📍 Context · OpenAI introduced "reasoning-token clustering" in GPT-5.5 Codex to reduce token usage and improve speed. But the community found this clustering sometimes breaks code context coherence, leading to lower quality output than older versions. Symptoms: wrong variable names, logical jumps, missing key context from earlier lines. This isn't a simple bug — it's a side effect of an optimization trade-off.

Why this matters

  • Directly affects your daily Codex usage — You use Codex every day for scripts and data work. If output quality drops, you'll spend significantly more time debugging and fixing.
  • Hidden cost increase — Lower quality means more iterations (more token consumption), potentially making Codex's claimed "efficiency gains" a net loss.
  • Community is already finding workarounds — Including downgrading to pre-GPT-5.5 versions, adjusting prompt structure. You need to know these strategies.
🎯 What you can do today
  1. Check your Codex version. If on latest GPT-5.5, try explicitly specifying an older model version (if API supports rollback).
  2. Read the HN discussion on GPT-5.5 Codex reasoning-token clustering for specific degradation cases.
  3. Add a quality check step to your workflow: after Codex output, do a quick review with another model (e.g., Fable 5).
