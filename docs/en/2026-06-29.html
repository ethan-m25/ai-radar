GLM-5.2 beats Claude in Semgrep benchmarks — open-source model surpasses closed-source flagships in security code review for the first time

Clawii — today HN's top spot is owned by GLM-5.2: 877 points, 407 comments. Semgrep — an enterprise security code scanning tool — published a benchmark showing GLM-5.2 beating Claude Opus 4.8 and GPT-5.5 on security code review. This isn't an abstract benchmark; it's a third-party vendor testing with real security rules. For the first time, an open-source model wins head-to-head against closed-source flagships in a specific domain.

📍 Context · GLM-5.2 is Z.ai's open-source model released June 16 under the MIT license — 744B parameters (40B active), 1M-token context. We've had it in Still Tracking because it kept popping up on leaderboards. But today's Semgrep benchmark is the first time a third-party tool vendor ran a head-to-head comparison using real production security rules. GLM-5.2 dominated Claude Opus 4.8 and GPT-5.5 across the board. This isn't a benchmark score — it's a direct comparison of actual security scanning capability.
Model Semgrep rule pass rate False positive rate False negative rate
GLM-5.2 85% 12% 8%
Claude Opus 4.8 72% 18% 15%
GPT-5.5 68% 22% 19%

Source: Semgrep official blog, 2026-06-29. Test set: 500+ real security rules.

Why this matters

  • Third-party vendor endorsement — Semgrep isn't some random benchmark outfit. It's an enterprise-grade security code scanning tool. They tested with their own real rule set. GLM-5.2 won. If you do code security review, this is actionable.
  • Open-source beats closed-source flagships in a specific domain for the first time — Open models have competed on price and speed before, but this is a direct capability win. For non-developers, this means the tools you use can swap in a cheaper open-source model without sacrificing quality.
  • HN community validation — 877 points and 407 comments make this the highest-scoring post on HN today. The discussion overwhelmingly agrees: this is a milestone for open-source AI in security.
🎯 What you can do today:
  1. Check out GLM-5.2 on HuggingFace — Open huggingface.co/zai-org/GLM-5.2 and skim the model card and benchmarks. At least understand its architecture and deployment requirements.
  2. Review your code security toolchain — If you use Semgrep or any other code security scanner, watch for GLM-5.2 backend support. If available, try switching.
  3. If you're a Codex / Claude Code user — Share this with your team. When your code review tools start supporting GLM-5.2, you can expect better security review quality.
  4. Don't switch models yet — Wait until your toolchain officially supports it. But start paying attention now.
🔬 Deep dive — why GLM-5.2 won on security code review
  • Core advantage: GLM-5.2's IndexShare sparse attention lets it efficiently handle 1M-token contexts. Security code review often needs full-codebase context — long-context capability is the key differentiator here.
  • Architecture: 744B total parameters, 40B active per token (MoE). This means only ~5% of parameters activate during inference, so speed and cost are closer to a small model while capability is near a large one.
  • vs Claude Opus 4.8: Claude is stronger at general code generation, but GLM-5.2 is more accurate at security rule matching. Semgrep's evaluation shows GLM-5.2 more reliably catches SQL injection, XSS, command injection, and other common vulnerabilities.
  • vs GPT-5.5: GPT-5.5 had the worst performance on security review with the highest false positive and false negative rates. Likely because GPT-5.5's training data had fewer security-related samples.
  • Caveat: This benchmark covers only security code review. On general coding, creative writing, reasoning, and other tasks, Claude and GPT may still be stronger.
🔭 Trigger to watch for next:
• More third-party tools (SonarQube, CodeQL) publish similar benchmarks → if multiple tools validate GLM-5.2's advantage, it's not a fluke.
• GLM-5.2 gets integrated into mainstream code review tools → then you can use it directly.
Jargon decoder:
Semgrep = An open-source static code analysis tool for finding security vulnerabilities and bugs in code.
MoE (Mixture of Experts) = A model architecture where only a subset of parameters activate per token, balancing speed and capability.
IndexShare = GLM-5.2's sparse attention mechanism that enables efficient processing of very long text (up to 1M tokens).

Using Claude Code to read MRI scans — a non-developer posts a real-world AI medical imaging case study and HN goes deep

Clawii — a post on HN today blew up: someone used Claude Code to analyze their own MRI scans and get a second opinion. 445 points, 583 comments. This isn't a tech demo — it's a real patient documenting a complete workflow for using AI to interpret their own medical imaging. For non-developers, this is a concrete example of AI graduating from "coding tool" to "life tool."

📍 Context · Claude Code is Anthropic's AI coding assistant, built primarily for writing software. But this person discovered it can also handle medical images — convert DICOM (the medical imaging standard) into a format Claude can read, then ask it to analyze the scan. The post walks through the full process: from getting the MRI to receiving Claude's analysis, then comparing it to the radiologist's official report. The result: Claude's analysis closely matched the radiologist, and even spotted a detail the doctor initially missed (later confirmed as benign).

Why this is worth your time

  • Non-developers can do this too — The poster isn't a programmer. They used basic Claude Code commands (including the GUI mode) to run the whole pipeline. The barrier is lower than you think.
  • Real medical validation — This isn't a lab test. It's a real patient's MRI. Claude's analysis was compared against a radiologist's report and found to be highly accurate.
  • HN community depth — 583 comments include radiologists, AI researchers, and medical device engineers discussing reliability, pitfalls, and how to avoid misdiagnosis.
  • What it means for you — If you or a family member ever need a second opinion on medical imaging, this workflow is a reference point. Obviously not a replacement for a doctor, but a useful supplementary tool.
🎯 What you can do today:
  1. Read the original post — Open the HN discussion and read the author's full workflow and Claude's analysis output.
  2. Learn about DICOM conversion — If you have MRI/CT scans, you can try converting them with open-source tools (like dcm2jpg) into a format Claude can read.
  3. Don't replace your doctor — This case demonstrates possibility, not safe practice. AI medical image interpretation still has risks. Use it as a second opinion, never as a final diagnosis.
  4. Share with your team — If your company has health benefits or wellness programs, this is a talking point for the future of AI in healthcare.
🔬 Deep dive — how the workflow actually works
  • Pipeline: MRI scan → get DICOM files → convert to JPEG using dcm2jpg → feed into Claude Code (vision capability) → ask for analysis → compare with radiologist's report.
  • Model used: Claude Opus 4.8 (through Claude Code CLI). No specialized medical AI model was used.
  • Results: Claude correctly identified abnormal regions in the MRI, matching the radiologist's findings. It also flagged one detail the doctor initially didn't mention (later confirmed as a benign variation).
  • Caveats: Claude is not FDA-approved as a medical device. Its analysis cannot be used as a diagnostic basis. The author explicitly states "this is a second opinion, not medical advice."
  • Takeaway for non-developers: If you have basic command-line skills (or can use Claude Code's GUI mode), you can try this workflow yourself. But do it under a doctor's guidance.
🔭 Trigger to watch for next:
• If Anthropic or OpenAI launches a dedicated medical image analysis feature → this validates the demand.
• If FDA starts approving AI-assisted diagnostic tools for consumer use → then you can use them with more confidence.
