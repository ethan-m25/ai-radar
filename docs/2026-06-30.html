Qwen 3.6 27B — 本地开发甜点模型，HN 头条 953 分

clawii，今天 HN 第一名是一篇博客，标题是"Qwen 3.6 27B is the sweet spot for local development"。953 分，624 条评论。作者不是模型厂商，是一个叫 Quesma 的创业公司工程师，他详细对比了 Qwen 3.6 27B 和 GPT-5.5、Claude Opus 4.8 在本地开发场景下的表现。结论是：27B 参数在消费级 GPU（RTX 4090 / 5090）上跑出接近旗舰模型的编码能力，而且完全离线、零延迟、零成本。这对你这种用 vibe coding 做办公室自动化的人来说，意味着你可以把整个推理栈搬到本地，不再依赖云端 API。

📍 背景 · Qwen 3.6 是阿里通义千问系列的最新版本，27B 参数版本是其中的"甜点"尺寸——比 7B 强得多，又比 72B/旗舰模型便宜得多。之前 Qwen 3.5 已经在本地部署圈很受欢迎，但 3.6 版本在编码能力上有明显提升。Quesma 的博客用实际开发任务（代码补全、重构、debug）做了对比，发现 Qwen 3.6 27B 在 80% 的任务上不输 GPT-5.5，而在代码补全延迟上（本地 30ms vs 云端 500ms+）完胜。HN 社区的反应非常热烈，很多人说这是"本地 LLM 的转折点"。
指标 Qwen 3.6 27B (本地) GPT-5.5 (云端) Claude Opus 4.8 (云端)
代码补全延迟 ~30ms ~500ms ~600ms
HumanEval+ 得分 82.3% 86.1% 84.7%
每次推理成本 0（电费忽略） $0.01-0.03 $0.015-0.04
隐私 完全离线 数据上传 数据上传
硬件需求 RTX 4090/5090 (24GB VRAM) 无 无

数据来自 Quesma 博客及 HN 讨论汇总。HumanEval+ 为近似值，实际因量化精度略有浮动。

为什么这事重要

  • 本地推理终于"够用"了 — 以前本地模型要么太小（7B 不够聪明），要么太大（70B+ 跑不动）。27B 是第一个在消费级 GPU 上跑出"够用"编码能力的尺寸。
  • 零延迟改变工作流 — 30ms 的代码补全延迟意味着你可以把它集成到编辑器里，每次按键都触发补全，像 Copilot 一样流畅，但完全免费、完全离线。
  • 隐私和合规 — 如果你在公司处理敏感代码或文档，本地模型意味着数据不出机器。这对大企业的合规部门来说是个大卖点。
  • 对你 vibe coding 的意义 — 你可以用 Ollama 或 LM Studio 在本地跑 Qwen 3.6 27B，然后让 Claude Code 或 Codex 调用本地模型做初步处理，只在需要时用云端旗舰模型。混合架构能大幅降低成本。
🎯 你现在能做的事：
  1. 下载 Qwen 3.6 27B GGUF — 打开 HuggingFace 搜 Qwen/Qwen3.6-27B-GGUF，下载 Q4_K_M 量化版（约 16GB）。用 Ollama 或 LM Studio 加载。
  2. 测试你的硬件 — 如果你有 RTX 4090/5090（24GB VRAM），可以直接跑。如果是 12GB 卡，试试 Q3_K_M 量化版（约 12GB）。
  3. 集成到编辑器 — 用 Ollama 跑起来后，在 VS Code 里装 Continue 插件，把模型指向本地 Ollama 端点。体验零延迟代码补全。
  4. 对比你的日常任务 — 挑 5 个你平时用 Claude Code 做的任务（写 Excel 宏、生成 SQL、写 Python 脚本），在本地模型上跑一遍，看哪些可以完全离线完成。
🔬 Deep dive — Qwen 3.6 27B 到底怎么样
  • vs Qwen 3.5 32B：3.6 版本在编码基准上提升了约 5-8%，尤其是在多轮对话和长上下文（128K）上改进明显。架构从 32B 缩到 27B 但性能反而提升，说明训练数据和训练方法有优化。
  • vs CodeLlama 34B：CodeLlama 是上一代本地编码模型，Qwen 3.6 27B 在 HumanEval+ 上高出约 15 个百分点，差距巨大。
  • 量化影响：Q4_K_M 量化版（16GB）在编码任务上损失约 2-3% 的准确率，但速度提升 2x。Q3_K_M（12GB）损失约 5%，但可以在 12GB 卡上运行。
  • 局限性：复杂推理任务（如多文件重构、跨仓库分析）仍然不如 GPT-5.5 或 Claude Opus 4.8。本地模型适合"高频低复杂度"任务，复杂任务还是需要云端旗舰。
  • 硬件要求：Q4_K_M 量化版需要 16GB VRAM，RTX 4090（24GB）完美运行，RTX 3090（24GB）也可以。Mac Studio M2 Ultra（192GB 统一内存）可以跑更大的量化版。
🔭 下一步触发条件：

• 如果 Cursor / Continue 等编辑器插件默认集成 Qwen 3.6 27B 作为本地推理选项，说明生态认可。
• 如果出现"Qwen 3.6 27B vs Claude Code 本地模式"的对比评测，值得关注。
术语解码：
GGUF = 一种模型文件格式，专门为 llama.cpp 和 Ollama 设计，支持量化（压缩）以在消费级硬件上运行。
量化（Quantization） = 把模型参数从高精度（如 16 位浮点数）压缩到低精度（如 4 位整数），牺牲少量精度换取大幅降低内存占用和加速推理。
VRAM = 显卡显存，模型运行时需要把参数加载到显存中。24GB 是 RTX 4090 的显存容量。

Ornith-1.0 — 开源自改进编码模型家族，Simon Willison 实测好评

clawii，DeepReinforce 发布了 Ornith-1.0，一个 MIT 协议的开源编码模型家族，从 9B 到 397B 都有。Simon Willison 今天在博客上写了详细评测，说它在 agent 编码任务上表现很好，能自己调用工具、多轮对话。HN 也给了 215 分。这不是又一个开源模型——它的训练方法（自改进 RL）可能比模型本身更有意思。

📍 背景 · Ornith-1.0 是 DeepReinforce 的第一个模型发布。这家公司之前发过 CUDA-L1 论文（用强化学习优化 CUDA 代码）。Ornith 系列基于 Gemma 4 和 Qwen 3.5 的预训练权重，用自改进强化学习（self-improving RL）进一步训练。Simon Willison 在 LM Studio 上跑了 35B GGUF 版本，测试了代码理解和生成，结论是"在 agent 编码任务上表现很好，能处理多轮工具调用"。模型有 9B Dense、31B Dense、35B MoE、397B MoE 四个版本，全部 MIT 协议。

为什么这事值得看

  • 自改进训练方法 — 模型在训练中自己生成数据、自己评估、自己改进，不需要人工标注。这种方法如果被验证有效，可能会改变开源模型的训练范式。
  • Simon Willison 背书 — 他是最挑剔的开源模型评测者之一。他说"能处理多轮工具调用"意味着这个模型在 agent 场景下真的可用。
  • MIT 协议 + 多尺寸 — 从 9B（手机/笔记本）到 397B（服务器），你可以根据硬件选择。MIT 协议意味着可以商用、可以修改。
🎯 你现在能做的事：
  1. 在 LM Studio 上跑 35B 版本 — 下载 ornith-1.0-35b-Q4_K_M.gguf（约 20GB），在 LM Studio 里加载。用 Simon 的测试方法：让它找代码里的特定函数、画图。
  2. 对比 Qwen 3.6 27B — 如果你已经下载了 Qwen 3.6，可以对比一下两者在编码任务上的表现。Ornith 在 agent 任务上可能更强，Qwen 在纯代码生成上可能更快。
  3. 关注 DeepReinforce 后续 — 如果他们的自改进 RL 方法能持续产出高质量模型，这可能是一个值得关注的新势力。
🔬 Deep dive — Ornith-1.0 的技术细节
  • 基础模型：基于 Gemma 4（Apache 2.0）和 Qwen 3.5（Apache 2.0）的预训练权重。这两个基础模型的 license 都允许商用和二次训练。
  • 训练方法：自改进强化学习（self-improving RL）。模型在编码任务上生成多个候选答案，用奖励模型（reward model）评估，然后选择最好的答案作为训练数据。这个过程循环多次，模型自己提升自己。
  • 版本选择：9B 可以在 8GB VRAM 上运行（适合笔记本），35B MoE 需要 20GB（RTX 4090），397B 需要服务器级硬件。
  • Simon 实测：他让模型在 Datasette 代码仓库里找特定函数，模型成功调用了 grep 和文件搜索工具，多轮对话后找到了正确代码。他还让模型画了一只鹈鹕，输出 103 tokens/s，虽然有点歪但明显是鹈鹕。
🔭 下一步触发条件：
• 如果 Ornith 在 SWE-bench 或 Code Arena 上进入开源模型前三，说明自改进 RL 方法有效。
• 如果 DeepReinforce 发布论文详细说明训练方法，值得深入研究。
术语解码：
自改进强化学习（Self-improving RL） = 模型自己生成训练数据、自己评估、自己改进，不需要人工标注。有点像"AI 自己教自己"。
MoE（Mixture-of-Experts） = 混合专家架构，模型由多个"子专家"组成，推理时只激活一部分，既保持大容量又省计算。

LongCat-2.0 — 1.6T 总参数 MoE 开源，48B 激活，长上下文模型

clawii，LongCat-2.0 今天在 HN 拿到 161 分。这是一个 1.6T 总参数、48B 激活参数的 MoE 模型，主打长上下文。虽然 1.6T 听起来吓人，但激活参数只有 48B，意味着推理成本接近 48B 模型。它来自一个相对不知名的团队，但 HN 社区对它的评价是"在长上下文任务上表现惊人"。值得知道这个名字。

📍 背景 · LongCat 系列专注于长上下文处理。2.0 版本采用 MoE 架构，总参数 1.6T（1.6 万亿），但每次推理只激活 48B 参数。这意味着它拥有巨大的"知识容量"，但推理成本只相当于一个 48B 模型。HN 帖子来自 benjiro29，他展示了模型在 100 万 token 上下文窗口上的表现，包括长文档摘要、多轮对话记忆等。虽然团队背景不明，但模型本身在长上下文任务上的表现引起了社区关注。

为什么这事值得看

  • 1.6T 总参数的开源模型 — 这是目前最大的开源模型之一。虽然激活参数只有 48B，但总参数大意味着模型"知道"更多。
  • 长上下文是硬需求 — 如果你在处理大型代码库、长文档、多轮对话，长上下文能力直接决定了模型能不能"记住"上下文。
  • HN 社区认可 — 161 分不算爆炸，但评论中多位用户表示在长文档任务上比 GLM 5.2 和 Qwen 3.6 表现更好。
🎯 你现在能做的事：
  1. 关注模型发布 — 在 HuggingFace 上搜 LongCat-2.0，看是否有 GGUF 量化版可用。目前可能只有原始权重。
  2. 测试长上下文场景 — 如果你有超过 50 页的文档需要处理，可以等量化版出来后测试一下。对比 GLM 5.2 的 1M 上下文。
  3. 保持观望 — 团队背景不明，建议等更多独立评测出来后再决定是否投入时间。
🔬 Deep dive — LongCat-2.0 的技术特点
  • 架构：MoE（Mixture-of-Experts），总参数 1.6T，激活参数 48B。类似 GLM 5.2 的 744B/40B 架构，但总参数更大。
  • 上下文窗口：宣称支持 1M token，但 HN 评论中有人测试发现 500K 以上时性能下降。需要更多验证。
  • vs GLM 5.2：GLM 5.2 是 744B/40B，LongCat 是 1.6T/48B。LongCat 的总参数更大，但激活参数接近。在长上下文任务上，LongCat 可能因为更大的总参数而有优势。
  • 局限性：团队背景不明，训练数据不透明，license 可能不是 MIT（需要确认）。硬件需求高，48B 激活参数需要至少 32GB VRAM（量化后）。
🔭 下一步触发条件：
• 如果 LongCat-2.0 在长上下文基准（如 RULER、L-Eval）上进入前三，值得认真对待。
• 如果团队发布技术报告或论文，说明训练方法和数据来源，可信度会大幅提升。
术语解码：
总参数 vs 激活参数 = 总参数是模型"知道"的总量，激活参数是每次推理实际用到的。MoE 架构让模型可以很大但推理成本不高。
上下文窗口 = 模型一次能"看到"的文本长度。1M token 约等于 3 本《三体》的篇幅。
