"用 AI 写更好的代码，写慢一点"

Nolan Lawson 的 HN 爆文（698 pts, 266 comments）——不是工具，不是模型，是一个用过同样工具的过来人在反思怎么才能真正用好它们。你每天用 Claude Code / codex，但你认真想过你的 workflow 对吗？这篇文章会逼你想。

📍 背景 · Context

Nolan Lawson —— Salesforce 的工程师，之前是微软 Edge 浏览器的核心开发者。他过去一年高强度使用 AI 写代码，最后得出的结论不是"AI 不够强"，而是"我用 AI 的方式不对"。他的核心观察：AI 让你写代码更快，但让你理解代码更慢。如果你只需要 generate 得快，短期爽，长期埋坑。

这不是反对 AI 编程的文章。这是批评"vibe coding 不加思考"的文章。 和你的日常直接相关。

EN

Why this matters — the single most practical thing you'll read this week

  • The thesis, in one sentence: AI makes you faster at producing code but slower at understanding code. The net effect is that you ship more code but with a growing "understanding debt" — code you wrote but don't fully grasp. This debt compounds.
  • Three specific traps Lawson calls out that you've definitely felt if you vibe-code daily:
    1. Premature generating — you jump to "let AI write it" instead of spending 5 minutes thinking about the approach. You save 5 minutes up front, lose 30 minutes later debugging.
    2. Habit erosion — AI writes your loops and conditionals for you so often that your ability to reason about control flow gets rusty. You start accepting AI-generated code without really checking it.
    3. The autocomplete brain — you start tab-completing through entire functions without reading them. The code works (usually), but you lose the muscle of understanding the codebase.
  • The counter-strategy he proposes: "Write better prompts, not more prompts." Spend more time describing what you want in natural language, review every line AI generates, and — counter-intuitively — take breaks from AI to write code from scratch to stay sharp.
  • Why this isn't just another opinion piece: 698 HN points in 11 hours with 266 comments means the developer community — people who use these tools every day — is actively wrestling with this. It's not a theoretical debate; it's people sharing what's actually working (and not working) in real codebases.
🎯 实操 — 你这周能做什么
  1. 花 20 分钟读原文：nolanlawson.com 直接打开。不是推销，不卖课，就是一个资深工程师的反思。读完你会对你自己每天用的工具产生新的看法。
  2. 做一次"你今天写的 AI 代码审计"：挑你今天用 Claude Code 或 codex 写的一段代码（任何长度）。逐行读一遍，问自己三个问题：(1) 我真的理解每一行在做什么吗？(2) 如果下周回来改这个文件，我能直接上手吗？(3) 有没有更简单的方案？
  3. 明天试一次"写 prompt 5 分钟，写代码 0 分钟"：在写任何一段代码之前，先花 5 分钟写/改 prompt——描述数据结构、边界情况、你预期的输出格式。然后让 AI 生成。对比你平时"写 30 秒 prompt → 生成 → 修 bug"的循环。记下差异。
  4. 每周选 1 天不用 AI：手动写一小段代码（哪怕只是一个脚本）。保持手感。Lawson 特别强调：你手动写代码的能力是你的"基本盘"——AI 让它生锈了没关系，但不能让它消失了。
🔬 深扒 — 为什么这篇文章能上 HN 头条 + 266 条讨论

这篇文章不是一个"AI 坏 / AI 好"的立场辩论。 它是在说：我们太关注"AI 能做什么"，太少关注"用 AI 的人应该怎么改变习惯"。

常见 vibe coding 方式Lawson 建议的方式
写 prompt 时间10-30 秒2-5 分钟（写之前先想清楚结构）
生成后动作merge 然后继续下一个逐行 review，不理解就追问 AI
犯错后反应让 AI 修 → 再试先理解为什么错，再决定怎么修
对自己代码的理解"大概知道它在做什么""每行都知道为什么在这里"
长期风险理解债务 + 维护困难知识积累 + 可控的代码库
适用场景原型、一次性脚本生产代码、长期维护

为什么这对你特别重要：你每周用 Claude Code 写很多代码。大部分情况下它工作得很好。但如果你只做"写 prompt → 生成 → merge → 下一个"的循环，你正在积累 Lawson 说的"理解债务"。这不像 bug 那样会立刻让你痛苦——它会在 3 个月后你回来改这个文件时狠狠地咬你一口。

我不是说每段代码都要逐行 review。脚本、原型、一次性任务 —— 大量生成完全没问题。但你的核心工作（薪酬分析模型、政策对比工具、重复性数据任务）总有一天需要回去维护。那些代码需要你理解。

一个诚实的 caveat：这篇文章的观点不是新东西。做 code review、理解自己写的代码——这些是软件工程的基本功。作者只是把它们放到了"AI 编程"这个新语境里重新讲了一遍。但正因为太基础，大部分人反而没在 AI 时代认真想过。

为什么这事重要

  • 核心观点一句话：AI 让你写代码更快，但让你理解代码更慢。如果不主动对抗这个副作用，你写的代码越多，未来欠的"理解债"越多。
  • 三个你肯定踩过的坑：(1) 不想就让它写——节约 5 分钟思考，亏掉 30 分钟 debug；(2) 习惯让它写所有循环和条件——你的逻辑推理能力在生锈；(3) tab auto-complete 到底——代码能用，但下周回来改的时候完全不记得它怎么工作的。
  • 实操要点：不是让你放弃 AI 编程——是让你在关键任务上主动"慢下来"。明天试一次：写 prompt 花 5 分钟描述清楚结构，再让 AI 生成。对比你平时的"30 秒 prompt → 生成 → 修 bug"循环。
