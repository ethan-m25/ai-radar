Kimi K3 — 2.8T 参数的开放前沿模型，7 月 27 日开源

爪哩，今天 Moonshot AI 正式发布了 Kimi K3，一个 2.8T 参数（2.8 万亿）的开放权重模型，1M token 上下文窗口，原生多模态（文本+图片+音频），承诺 7 月 27 日前开源。HN 1679 点，989 条评论，是今天 HN 上最热的 AI 话题。K3 在 Frontend Code Arena 上以 76% 的胜率领先，排名高于 Claude Fable 5 和 GPT-5.6 Sol。它还引入了 Kimi Delta Attention（KDA）技术，解码速度提升 6.3 倍。

📍 背景 · Moonshot AI 是 Kimi 的母公司，之前发布了 Kimi K2.7，已经在编码 agent 社区有不错的口碑。K3 是他们的旗舰模型，2.8T 参数是公开可用的最大模型之一（仅次于 GPT-5.6 系列）。开放权重承诺 7 月 27 日前发布，这对开源社区来说是一个重大事件。K3 在 Frontend Code Arena 以 76% 胜率领先，说明它在前端编码任务上非常强——这对 vibe coding 用户来说是好消息。

为什么这事重要

  • 2.8T 参数的开放权重模型 — 这是目前公开可用的最大开放权重模型之一。总参数 2.8T，虽然具体活跃参数未公开，但大概率是 MoE 架构。对爪哩来说，这意味着你在编码 agent 中有了一个新的强模型选择。
  • Frontend Code Arena 76% 胜率 — 在前端编码任务上，K3 胜率高于 Claude Fable 5 和 GPT-5.6 Sol。这对做 Web 开发的 clawii 来说是好消息。
  • KDA 技术实现 6.3x 解码加速 — 虽然这与用户无关，但意味着推理成本更低、速度更快。
  • 7 月 27 日开源承诺 — 如果按承诺发布，这将是一个 Apache 2.0 或类似协议的开放权重模型，对生态是重大利好。

🔬 深扒——Kimi K3 和其他开放模型有什么不同？
方面说明
核心思路2.8T 总参数，1M token 上下文窗口，原生多模态（文本+图片+音频）。引入了 Kimi Delta Attention（KDA）实现 6.3x 解码加速，以及 Attention Residuals 实现约 25% 的训练效率提升。
vs GLM-5.2（753B）K3 总参数更大（2.8T vs 753B），但 GLM-5.2 在中文任务上可能有优势。K3 在 Frontend Code Arena 胜率更高（76% vs 未公开）。两者都是开放权重，K3 承诺 7 月 27 日开源。
vs Hy3（295B）Hy3 是 295B 总参数 / 21B 活跃，K3 参数更多。Hy3 有 192 个专家，K3 的架构细节未完全公开。K3 在编码基准上更强。
vs Inkling（975B / 41B 活跃）Inkling 是美国团队的开放模型，K3 是中国的。K3 总参数更大，但 Inkling 已经开源（Apache 2.0），K3 承诺 7 月 27 日开源。两者在多模态和编码能力上各有优势。
技术实现MoE 架构（推测），KDA 注意力机制，Attention Residuals 训练优化。支持 1M 上下文。
上手难度低。可以通过 kimi.com 网站或 API 直接使用。开源后可以本地部署（需要多 GPU）。
诚实警告K3 在整体用户体验上仍落后于 Fable 5 和 GPT-5.6 Sol（独立评测）。另外，2.8T 参数意味着完整部署需要大量 GPU。开源承诺是 "7 月 27 日"，但具体协议和开源方式还未确认。

老实说：Kimi K3 是一个值得关注的模型，但不要被 2.8T 参数吓到——MoE 架构意味着每次推理只激活部分参数，实际成本可控。它在 Frontend Code Arena 上表现亮眼，对做前端开发的 clawii 来说是一个有力的新选择。但独立评测显示它在整体体验上仍落后于 Fable 5 和 GPT-5.6 Sol，所以不要期待它全面超越。

🔭 关注点 · 关注 7 月 27 日的开源发布——具体协议、模型大小、可用平台。同时关注 K3 是否被 Cursor、Codex、Claude Code 等编码 agent 工具集成。如果集成，说明它已经达到了实用水平。
术语解码 · KDA（Kimi Delta Attention） = Kimi Delta Attention，Moonshot AI 提出的注意力机制优化，实现 6.3x 解码加速。Attention Residuals = 注意力残差连接，一种训练优化技术，提升约 25% 的训练效率。Frontend Code Arena = 前端编码竞技场，一个评估模型前端编码能力的基准测试。

LM Studio Bionic — 本地 AI agent 运行开放模型

爪哩，LM Studio 发布了 Bionic，一个可以在本地运行开放模型的 AI agent。HN 264 点，91 条评论。这意味着你可以在自己的电脑上运行一个 AI agent，所有数据都在本地处理，不需要上传到云端。对隐私敏感的 clawii 来说，这是一个值得关注的选择。

📍 背景 · LM Studio 是一个本地运行开源大模型的桌面工具，支持 macOS、Windows、Linux。它使用 GGUF 格式的模型，可以下载 HuggingFace 上的模型并在本地运行。Bionic 是 LM Studio 的 AI agent 功能，让用户可以在本地运行 agent 任务——代码生成、文件操作、Web 搜索等。之前本地 agent 的选项很少（主要是 Ollama + 一些实验性工具），LM Studio 的加入让这个领域更成熟了。

为什么这事重要

  • 本地运行 agent，无需上传数据 — 对于处理敏感数据的 clawii 来说，这是一个真正的隐私保护方案。所有数据都在本地处理，不会泄露到云端。
  • 支持开放模型生态 — Bionic 可以运行 HuggingFace 上的任何 GGUF 模型，包括 GLM-5.2、Qwen、Hy3 等。这意味着你可以选择最适合你的模型。
  • 降低了本地 agent 的使用门槛 — LM Studio 的界面友好，不像命令行工具那样需要技术背景。下载安装即可使用。

🔬 深扒——LM Studio Bionic 和 Claude Code / Codex 有什么不同？
方面说明
核心思路一个本地运行的 AI agent，支持代码生成、文件操作、Web 搜索等任务。使用 HuggingFace 上的 GGUF 模型，所有数据在本地处理。
vs Claude Code / CodexClaude Code 和 Codex 都是云端 API，你的代码会上传到 Anthropic/OpenAI 的服务器。Bionic 完全本地运行，数据不出电脑。但 Bionic 的模型能力受限于本地硬件的性能——你无法在本地运行 975B 的 Inkling 或 2.8T 的 K3。
技术实现基于 GGUF 格式的量化模型，支持 CPU 和 GPU 推理。使用 llama.cpp 作为推理后端。Bionic 的 agent 功能包括代码执行、文件操作、Web 搜索等。
上手难度低。下载安装即可使用，不需要命令行。但需要下载大模型文件（几个 GB 到几十 GB）。
诚实警告本地运行的模型能力有限。你无法在本地运行 2.8T 的 K3 或 975B 的 Inkling。Bionic 适合运行 7B-70B 的模型。对于复杂任务，云端 agent 仍然更强大。

老实说：LM Studio Bionic 是一个很好的本地 agent 选项，但不要期待它能替代 Claude Code 或 Codex。本地模型的能力天花板就在那里。Bionic 的真正价值在于隐私保护——对于处理敏感数据的工作，它可以作为一个安全的选择。

🔭 关注点 · 关注 Bionic 的模型支持范围——它是否能运行最新的开放模型（如 GLM-5.2、K3 开源后）。另外，关注它的 agent 功能是否足够强大，能否处理复杂的多步骤任务。
术语解码 · GGUF = GPT-Generated Unified Format，一种模型量化格式，可以在本地硬件上高效运行大模型。llama.cpp = 一个用 C++ 编写的 LLM 推理引擎，支持 CPU 和 GPU 推理，是 LM Studio 的后端。

NotebookLM 更名为 Gemini Notebook

爪哩，Google 将 NotebookLM 更名为 Gemini Notebook，整合到 Gemini 品牌下。HN 315 点，160 条评论。NotebookLM 是 Google 的 AI 笔记工具，可以基于你上传的文档（PDF、网页、笔记）生成问答、摘要、播客等。更名为 Gemini Notebook 意味着它现在是 Gemini 产品线的一部分，可能有新的功能整合。

📍 背景 · NotebookLM 是 Google 在 2023 年推出的 AI 笔记工具，最初名为 Project Tailwind。它基于你上传的文档生成回答，支持语音对话（Audio Overviews）。2024 年更名为 NotebookLM，2025 年加入 Gemini 2.0 支持。现在更名为 Gemini Notebook，标志着它正式成为 Gemini 品牌的一部分。这通常意味着更深度的整合——比如与 Gemini 的 Web 搜索、代码执行、多模态能力的结合。

为什么这事重要

  • 品牌整合意味着功能升级 — 当一个产品被纳入主品牌，通常意味着它将获得更多资源和支持。Gemini Notebook 可能很快会获得 Gemini 的 Web 搜索、代码执行、多模态能力。
  • 对知识工作者有用 — NotebookLM 的核心功能是基于文档回答问题。对 clawii 来说，这意味着你可以上传公司文档（薪酬政策、员工手册等），然后让 AI 基于这些文档回答你的问题。
  • 可能替代 Google Docs 的 AI 功能 — 如果 Gemini Notebook 与 Google Docs 深度整合，它可能成为一个强大的文档 AI 助手。

🔬 深扒——Gemini Notebook 和 Notion AI / Obsidian AI 有什么不同？
方面说明
核心思路基于你上传的文档生成问答、摘要、播客。文档来源包括 PDF、Google Docs、网页、Markdown 等。支持多轮对话和引用来源。
vs Notion AINotion AI 是 Notion 的内置 AI 功能，直接在笔记中生成内容。Gemini Notebook 更专注于"基于文档的问答"，而不是内容生成。Notion AI 使用 GPT-4 系列，Gemini Notebook 使用 Gemini 系列。
vs Obsidian AIObsidian 的 AI 功能通过插件实现（如 Copilot），需要 API key。Gemini Notebook 是免费使用的（有使用限制）。Obsidian 的 AI 在本地运行（如果使用本地模型），Gemini Notebook 是云端。
技术实现基于 Gemini 2.5 或 3.0 系列。支持文档检索、多轮对话、引用来源。还支持 Audio Overviews（生成播客风格的对话）。
上手难度极低。打开网站，上传文档，开始提问。不需要任何配置。
诚实警告Gemini Notebook 仍然是一个"基于文档的问答"工具，不是一个通用的 AI 助手。它不能替代 Google Search 或 ChatGPT。另外，文档上传后存储在 Google 服务器上，注意隐私问题。

老实说：更名本身不是一个功能更新，更多是品牌整合。但 Google 通常会在更名后推出新功能，所以值得关注接下来的变化。对 clawii 来说，如果 Gemini Notebook 整合了 Gemini 的 Web 搜索和代码执行能力，它可能成为一个有用的知识管理工具。

🔭 关注点 · 关注 Gemini Notebook 是否获得新的功能——比如 Web 搜索、代码执行、多模态支持。另外，关注它是否与 Google Docs 和 Google Drive 更深度的整合。
术语解码 · Audio Overviews = NotebookLM 的一个功能，可以基于文档生成播客风格的对话音频，两个 AI 主持人讨论文档内容。RAG（Retrieval-Augmented Generation） = 检索增强生成，一种 AI 技术，在回答问题时先检索相关文档，然后基于检索结果生成回答。Gemini Notebook 的核心技术就是 RAG。
