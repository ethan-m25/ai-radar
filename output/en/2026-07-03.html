llm-coding-agent 0.1a0 — Simon Willison open-sources a coding agent framework

Clawii — Simon Willison built a lightweight coding agent using Fable 5, then packaged it as a Python library and pushed it to PyPI. This isn't a product — it's an agent framework you can pip install. If you've felt locked into Claude Code or Codex CLI, there's now an open-source, model-agnostic alternative you can hack on yourself.

📍 Context · Simon Willison is a well-known figure in the Python community (Django co-creator) and longtime maintainer of the llm library — a model abstraction layer that lets you swap LLM backends. He used Claude Code + Fable 5 to write the entire llm-coding-agent himself: he wrote the spec, then let the agent self-bootstrap the implementation. The project provides file read/edit, command execution, search, and file listing tools — all through the llm tool system. The fact that an agent wrote its own agent framework is the meta-signal here.

Why this matters

  • Open-source, model-agnostic — MIT-licensed. You can swap in Fable 5, GPT-5.5, Opus 4.8, or open-weight models via BYOK. If your company requires data sovereignty, this is the on-ramp.
  • Self-bootstrapping validates the category — The agent wrote itself. Simon wrote a spec, Claude Code built the whole thing. This means coding agent capability has reached the point where agents can produce production-quality agent frameworks.
  • One-liner to try — uvx --prerelease=allow --with llm-coding-agent llm code --yolo. No Cursor, no Claude Desktop install. Feels similar to Claude Code out of the box.
  • Simon's credibility — He's already using it in his own Datasette project. This isn't vaporware.
🎯 What you can do today:
  1. Run uvx --prerelease=allow --with llm-coding-agent llm code --yolo and give it a simple task (write a script, refactor a file).
  2. Read Simon's blog post and the spec file to understand the tool set and design philosophy.
  3. If you want to reduce Claude Code dependency, try configuring a local Fable 5 or GLM-5.2 backend through the llm library and test the agent with an open-weight model.
🔬 Deep dive — How this differs from Claude Code / Codex CLI
  • vs Claude Code: Claude Code is closed-source (free but proprietary). llm-coding-agent is MIT and lets you use any model backend. Claude Code has richer file+terminal+browser tools; this agent has a simpler tool set (read/edit/file/search/execute). Claude Code is polished; this is alpha.
  • vs Codex CLI: Codex CLI is OpenAI's closed-source CLI agent. llm-coding-agent is open-source and model-agnostic. Codex CLI ties you to GPT-5.x; this agent works with any llm-compatible backend.
  • What it runs on: Depends on Simon's llm library, which supports OpenAI, Anthropic, and open-weight models. Default picks up whatever API key is in your environment.
  • How hard to set up: Near-zero if you have Python and uv. One command to run. ~5 minutes.
  • Honest caveat: This is an alpha release. Tool set is basic — no multi-step planning, no deep context management. But as an open-source starting point, it proves the barrier to building your own agent is now extremely low.

Claude-real-video — 200-line script that lets any LLM watch video

Clawii — A 200-line Python script that splits video into frames + audio transcripts, then feeds them to any LLM for analysis. Works with Claude, GPT, Gemini. Not a product — a clever pattern. Use it to let your coding agent watch a UI walkthrough video or analyze a tutorial recording.

📍 Context · Most LLMs today can only handle text and images. True video understanding requires native multimodal support that few models have. Claude-real-video works around this by using ffmpeg to extract frames at intervals, Whisper for audio transcription, and then sending the frames as base64 images + text to the LLM in sequence. The approach isn't new — researchers have done this for months — but the packaging as a one-shot script that hit HN front page signals rising mainstream demand for "video as agent input."

Why this is worth your attention

  • Turns video into agent context — Record a UI walkthrough, then ask an agent to write code or answer questions based on what happened in the video. Useful for non-developers recording tutorials and auto-generating documentation.
  • Works with existing models — No need to wait for Claude or GPT to natively support video. You can use today's models with this workaround.
  • HN traction reveals demand — 139 points, 87 comments. The discussion is mixed (slow, token-heavy) but the activity itself shows people want this capability.
  • Bridge to native video understanding — When Claude or GPT eventually ship native video support (rumors are circulating), this approach becomes obsolete. Until then, it's a practical bridge.
🎯 What you can do today:
  1. Clone github.com/huangchihhungleo/claude-real-video and install dependencies (ffmpeg + Python packages).
  2. Find a short tutorial video (1-2 minutes). Run the script and see if the LLM correctly understands what happened.
  3. Estimate token cost: a 3-minute video at 1 frame per 2 seconds = ~90 images. At ~800 tokens per image, that's ~72K tokens per analysis. Check your API budget.
🔬 Deep dive — How it works
  • Step 1: ffmpeg extracts frames at fixed intervals (configurable, e.g., every 2 seconds) or via scene-change detection.
  • Step 2: Whisper transcribes audio to text with timestamps.
  • Step 3: Each frame (as base64) is paired with the corresponding transcript segment and sent to the LLM as a multimodal prompt.
  • Step 4: For long videos, segments are analyzed independently and then summarized.
  • What it runs on: Python + ffmpeg + Whisper. LLM calls go through standard APIs (Claude, GPT, Gemini).
  • How hard to set up: Moderate — ffmpeg installation, Python deps, API keys. A 5-minute video may take 10+ minutes to process.
  • Honest caveat: Token cost is real. Frame rate × tokens per image × number of segments adds up fast. Quality depends on frame selection — you miss things between frames. Google Gemini already natively supports video, so this is a workaround for models that don't.
🔭 Trigger to watch for next:
• If Claude or GPT announce native video input support (rumored), this tool becomes obsolete — but the demand signal is valuable.
• If the "screen recording → agent auto-operates" pattern gains traction, expect more polished tools that combine this approach with coding agents.
Jargon decoder:
ffmpeg = a ubiquitous open-source tool for processing video and audio files. Used here to extract frames.
Whisper = OpenAI's open-source speech-to-text model. Used here to generate video transcripts.
base64 = a way to encode binary data (like images) as text, so they can be sent in API calls.
