"Using AI to Write Better Code, More Slowly"

Nolan Lawson's HN front-page essay (698 pts, 266 comments) — not a tool, not a model release. A senior engineer reflecting on how to actually use AI coding tools well. You use Claude Code and codex every day. Have you thought about whether your workflow is making you better — or just faster?

📍 Context

Nolan Lawson is a staff engineer at Salesforce who previously worked on Microsoft Edge's browser engine. After a year of heavy AI-assisted coding, his conclusion is not "AI isn't strong enough" — it's "I was using AI wrong." His core observation: AI makes you faster at producing code but slower at understanding it. If you only care about generating fast, you feel good short-term and pay for it long-term. This is not an anti-AI coding piece. It's a critique of vibe coding without thinking — directly relevant to how you work every day.

Signal Analysis

Why this matters — the most practical thing you'll read this week

  • The thesis, in one sentence: AI makes you faster at producing code but slower at understanding it. The net effect: you ship more code but accumulate "understanding debt" — code you wrote but don't fully grasp. This debt compounds over time.
  • Three specific traps Lawson calls out that you've definitely felt if you vibe-code daily:
    1. Premature generating — you jump to "let AI write it" instead of spending 5 minutes thinking about the approach. You save 5 minutes up front, lose 30 minutes later debugging.
    2. Habit erosion — AI writes your loops and conditionals so often that your ability to reason about control flow gets rusty. You start accepting AI-generated code without really checking it.
    3. The autocomplete brain — you tab-complete through entire functions without reading them. The code works (usually), but you lose the muscle of understanding the codebase.
  • The counter-strategy he proposes: "Write better prompts, not more prompts." Spend more time describing what you want in natural language, review every line AI generates, and — counter-intuitively — take breaks from AI to write code from scratch to stay sharp.
  • Why this isn't just another opinion piece: 698 HN points in 11 hours with 266 comments means the developer community — people who use these tools every day — is actively wrestling with this. It's not a theoretical debate; it's people sharing what's actually working (and not working) in real codebases.
🎯 Action — what you can do this week
  1. Read the original post (20 min): nolanlawson.com — open it now. Not a sales pitch, not a course. A veteran engineer's honest reflection. You'll see your own daily tools differently afterward.
  2. Audit one piece of AI-generated code you wrote today: Pick any block of code you wrote with Claude Code or codex. Read it line by line. Ask yourself three questions: (1) Do I actually understand every line? (2) If I came back to this file next week, could I work on it immediately? (3) Is there a simpler approach?
  3. Tomorrow, try "5 minutes of prompt thinking, 0 minutes of promptless coding": Before writing any code, spend 5 minutes writing/refining your prompt — describe the data structures, edge cases, expected output format. Then let AI generate. Compare the result against your usual "30 seconds of prompt → generate → fix bugs" loop.
  4. Pick one day a week to code without AI: Write a small script or function from scratch. Keep the muscle. Lawson's point: your manual coding ability is your foundation — AI can let it get rusty, but it shouldn't let it disappear.
🔬 Deep dive — Why this essay landed on HN's front page with 698 points

This is not a "AI is bad / AI is good" position piece. It's saying: we've been so focused on what AI can do that we've ignored how our own habits need to change.

Why this hits close to home for you: You write a lot of code with Claude Code every week. Most of the time it works great. But if your loop is "write prompt → generate → merge → next," you're accumulating the "understanding debt" Lawson describes. It won't hurt today. It will hurt in 3 months when you come back to that file.

I'm not saying every line needs review. Scripts, prototypes, one-off tasks — blast through them. But your core work (compensation analysis models, policy comparison tools, repetitive data tasks) will eventually need maintenance. Those files need to be understood.

Honest caveat: None of this is new. Code review, understanding your own code — these are software engineering fundamentals. Lawson just reframed them for the AI era. But precisely because they're basics, most people haven't thought about them seriously post-AI.
