Better Models: Worse Tools — newer models get worse at custom tool calls

Clawii — Armin Ronacher, the creator of Pi, found that Opus 4.8 and Sonnet 5 hallucinate parameter names in custom tool schemas more frequently than older models. This isn't a random bug — Anthropic's RL training may have locked the new models' tool-calling behavior to Claude Code's native edit tools, breaking other frameworks (Pi, LangChain, etc.).

📍 Context · Coding agents depend on models correctly understanding tool schemas and generating valid arguments. Models have steadily improved at schema following over the past year, but Armin reports the opposite trend: Opus 4.8 and Sonnet 5 insert non-existent fields into the edit tool's edits[] array, causing tool calls to be rejected. Simon Willison independently verified and discussed this phenomenon on his blog.

Why this matters

  • Directly affects the tools you use — If you use any coding agent other than Claude Code (Pi, LangChain, a custom CLI), newer models are less reliable. You may need to downgrade to older models.
  • The root cause is worth watching — This isn't an accident. It's a consequence of RL training: models were specifically optimized to use Claude Code's built-in search-and-replace tool, and compatibility with other formats suffered.
  • Strong signal — HN front page #15, 180 points, 63 comments. Two front-line developers (Armin and Simon) are both saying the same thing. This isn't a one-off.
🎯 What you can do today
  1. Check if your coding agent framework uses Sonnet 5 or Opus 4.8 — if so, watch for increased tool call failure rates
  2. Consider model routing: use Opus 4.8 for complex reasoning, but downgrade to Sonnet 4 or Opus 4 for tool-call-heavy tasks
  3. Watch for updates from Pi, LangChain, and other frameworks — they'll likely adapt to the new model behavior shift soon

Honest take: This isn't the end of the world — you can always fall back to older models. But the trend is worrying: when base models are optimized for a closed toolchain, the whole ecosystem fragments.

GPT-5.5 Codex reasoning-token clustering causes performance degradation

Clawii — an HN post with 286 points reports that GPT-5.5 Codex output quality has degraded due to reasoning-token clustering. If you use Codex for coding, you may have noticed weird output lately — it's not your imagination.

📍 Context · GPT-5.5 Codex is OpenAI's primary coding agent model. Reasoning-token clustering is a speed optimization that groups similar reasoning tokens together for batch decoding. But the community has found that under certain conditions, the clustering algorithm causes token distribution drift, producing output with repeated or incoherent segments. This isn't a minor tweak — it's affecting a wide range of users.

Why this matters

  • If you use Codex, you may already be affected — weird blank lines, duplicate comments, or logical jumps in generated code. If you've ruled out prompt issues, the model likely changed.
  • The root cause is clear — it's not a benchmark micro-fix but a side effect of reasoning-token clustering. OpenAI may fix it soon, but short-term avoidance strategies matter.
  • Strong community reaction — 286 points is the second-highest AI-related score on HN today. 113 comments with many users posting their own failure cases.
🎯 What you can do today
  1. If you use Codex CLI or the Codex app, re-run a task you've previously completed and compare output quality
  2. Temporarily switch to an older model (GPT-5.5 without clustering or downgrade to 5.4) or use an alternative (Claude Code / Codex fallback)
  3. Check OpenAI's status page for updates
🔬 Deep dive — what is reasoning-token clustering and why does it cause degradation

Impact on coding: If you're writing a long function that requires precise logic, clustering can mis-group reasoning tokens, breaking code structure. Community reports include lost loop variables and misjudged conditional branches.

Honest take: This is an engineering trade-off. OpenAI chose clustering for speed, but for coding agents — where precision matters — it may not be worth it. Good news: it's likely a bug in the algorithm rather than permanent regression.

🔭 What to watch for · Will OpenAI ship a fix or provide a toggle? Are there community workarounds (e.g., adjusting temperature to change token distribution)? If the fix takes too long, it may accelerate model switching.
