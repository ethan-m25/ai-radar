Bonsai 27B — 一个能在手机上跑的 27B 级模型

爪哩，今天 HN 上有个项目炸了——Bonsai 27B，612 points，214 条评论。PrismML 把 Qwen 3.6 27B 用三值量化（ternary quantization）压缩到了一个可以在手机上本地运行的版本。不是云端调用，是真正在你的手机上跑。这意味着你可以在飞机上、地铁里、没有网络的地方，用手机处理文档、分析图片、甚至写代码。对 vibe coding 来说，这是一个 game changer——你的 AI 助手终于可以随身携带，不依赖网络了。

📍 背景 · 本地模型一直有个矛盾：参数越多越聪明，但参数越多越跑不动。27B 参数的模型通常需要至少 16GB 显存，手机根本不可能。PrismML 用三值量化（把权重从 16 位压缩到 1.58 位）把模型大小从 54GB 压缩到不到 4GB，同时保持大部分能力。这不是第一个手机端模型，但 27B 级别是第一次——之前的手机模型大多是 7B 或更小。Bonsai 27B 在多个基准测试上接近 Qwen 3.6 27B 原版，但体积只有 1/10。

为什么这事重要

  • 离线 AI 不再是玩具 — 之前的手机端模型（如 Gemma 2B、Phi-3）能力有限，只能做简单问答。Bonsai 27B 可以处理复杂的多模态任务——分析图片、理解文档、写代码。对爪哩来说，这意味着你可以在没有网络的地方让 AI 帮你处理工作。
  • 隐私和安全 — 所有数据都在本地处理，不会上传到云端。对于处理公司敏感数据（比如薪资表）的爪哩来说，这是一个巨大的优势。
  • 对 vibe coding 的直接影响 — 你可以在手机上用本地模型辅助编码，比如分析代码片段、生成注释、甚至运行简单的代码生成。虽然不如 Claude Code 强大，但胜在随时可用。

🔬 深扒——Bonsai 27B 和直接在云端跑 Qwen 3.6 有什么不同？
方面说明
核心思路Bonsai 27B 基于 Qwen 3.6 27B，使用三值量化（ternary quantization）将权重从 FP16 压缩到 1.58 位（每个权重只用 1.58 位表示），模型大小从 ~54GB 降到 ~4GB。同时保留了多模态能力（文本+图片输入）。
vs 云端 Qwen 3.6 27B云端版本更聪明（基准测试高 5-10%），但需要网络，有延迟，有隐私风险。Bonsai 27B 能力接近（在多个基准上达到原版的 85-95%），但完全离线、零延迟、隐私安全。
vs 其他手机端模型之前的手机模型（Gemma 2B、Phi-3、Qwen 2.5 7B）参数少，能力有限。Bonsai 27B 是第一个在手机上跑 27B 级模型的方案，能力大幅提升。
技术实现三值量化（ternary quantization）将权重限制为 {-1, 0, +1}，每个权重只需 1.58 位。配合高效的推理内核（如 llama.cpp 的量化支持），可以在手机 CPU/GPU 上实时运行。
上手难度中等。需要下载 GGUF 格式的模型文件（~4GB），然后用支持 GGUF 的推理应用（如 MLCChat、LM Studio）加载。对于非开发者，可能需要一些教程指导。
诚实警告三值量化会损失一些精度，在复杂推理任务上可能不如原版。手机运行速度取决于设备——旗舰手机可以流畅运行，中端手机可能较慢。另外，4GB 的模型文件对手机存储也是一个考验。

老实说：Bonsai 27B 不是用来替代 Claude Code 或 Codex 的。它的价值在于填补了一个空白——当你没有网络、或者不想把数据传到云端时，你仍然有一个足够聪明的 AI 助手。对于爪哩来说，最实用的场景可能是：在出差路上用手机处理文档、分析图片，或者在公司内网（不能连外网）的环境下使用。

🔭 关注点 · 如果 Bonsai 27B 的 GitHub 或 HuggingFace 下载量在一周内快速增长，说明社区认可度高。另外，关注它是否被主流推理框架（如 llama.cpp、MLCChat）原生支持，以及是否有针对特定任务（如编码、文档分析）的微调版本。
术语解码 · 三值量化（Ternary Quantization） = 一种模型压缩技术，把权重限制为 -1、0、+1 三个值，每个权重只需 1.58 位存储，大幅减小模型体积。GGUF = 一种模型文件格式，用于在 llama.cpp 等本地推理框架中加载量化模型。多模态 = 模型可以同时处理文本和图片输入。
