Inkling 975B — 美国最强开放权重模型，Apache 2.0 发布

爪哩，今天 Thinking Machines Lab 正式发布了 Inkling，一个 975B 参数（41B 活跃）的 MoE 模型，Apache 2.0 协议，支持 1M 上下文，多模态（文本+图片+音频）。Mira Murati、Soumith Chintala、John Schulman、Lilian Weng 都在推上背书。这是美国第一个真正能打的开放权重模型——之前美国开源模型一直落后中国（Hy3、GLM-5.2），Inkling 终于把差距缩小了。

📍 背景 · 开放权重模型领域一直是中国占优：腾讯 Hy3 295B、智谱 GLM-5.2 753B、DeepSeek V4 系列。美国这边，Meta 的 Llama 系列虽然用户多，但能力上已经被中国模型超越。Thinking Machines Lab 由前 OpenAI CTO Mira Murati 创立，团队包括 Soumith Chintala（PyTorch 创始人）、John Schulman（RLHF 发明人）、Lilian Weng（OpenAI 安全负责人）。Inkling 是他们第一个正式发布的模型，975B 参数但只有 41B 活跃（MoE 架构），所以推理成本可控。Apache 2.0 协议意味着你可以随意商用、修改、再发布。

为什么这事重要

  • 美国终于有了能打的开放权重模型 — 之前美国开源模型（Llama 4、Gemma 4）在编码和 agent 任务上落后中国模型 10-20%。Inkling 在多个基准上接近 Hy3 和 GLM-5.2，部分任务甚至领先。对爪哩来说，这意味着你可以用 Apache 2.0 的模型做商用项目，不用担心 license 问题。
  • Mira Murati + Soumith + Schulman 的组合 — 这不是一个普通创业公司。这些人做过 GPT-4、PyTorch、RLHF。他们的模型值得认真对待。
  • 1M 上下文 + 多模态 — 支持文本、图片、音频输入，1M token 上下文窗口。对 vibe coding 来说，这意味着你可以把整个代码库塞进去，让模型理解全局。

🔬 深扒——Inkling 和 Hy3 / GLM-5.2 有什么不同？
方面说明
核心思路975B 总参数，41B 活跃参数（MoE 架构）。支持文本、图片、音频输入，文本输出。1M token 上下文窗口。Apache 2.0 协议。
vs 腾讯 Hy3 295BHy3 是 295B 总参数 / 21B 活跃，256K 上下文。Inkling 参数更多（975B vs 295B），活跃参数也更多（41B vs 21B），上下文更大（1M vs 256K）。但 Hy3 有 192 个专家，Inkling 的专家数未公开。两者在编码基准上接近，Inkling 在多模态上更强（支持音频）。
vs 智谱 GLM-5.2 753BGLM-5.2 是 753B 总参数，活跃参数未公开。Inkling 总参数更大（975B），但 GLM-5.2 在中文任务上更强。两者都是 Apache 2.0。GLM-5.2 在 Coding Arena 上领先，但 Inkling 在多模态上更全面。
技术实现MoE（Mixture of Experts）架构，每次推理只激活部分参数（41B），所以推理成本可控。支持 vLLM、SGLang、Modal、Baseten、Databricks 等平台。
上手难度低。可以通过 API 直接调用（Tinker、Together 等平台），也可以自己部署（需要多 GPU 集群）。对于 vibe coding 用户，直接调 API 就行。
诚实警告Inkling 在部分基准上仍落后中国顶级模型（Hy3、GLM-5.2）。另外，975B 总参数意味着完整部署需要大量 GPU，不是个人能跑的。但通过 API 使用成本可控。

老实说：Inkling 的意义不在于它是不是最强模型——它可能不是。它的意义在于：美国终于有了一个 Apache 2.0 的、由顶级 AI 研究员打造的开放权重模型。这对整个开源生态是好事。对爪哩来说，这意味着你有了更多选择，不用只依赖中国模型或闭源 API。

🔭 关注点 · 关注 Inkling 在 Coding Arena 和 SWE-bench 上的正式成绩。另外，关注它是否被 Cursor、Codex、Claude Code 等编码 agent 工具集成。如果集成，说明它已经达到了实用水平。
术语解码 · MoE（Mixture of Experts） = 混合专家架构，模型有多个"专家"子网络，每次只激活部分专家，在保持大模型能力的同时控制推理成本。Apache 2.0 = 一种开源协议，允许商用、修改、再发布，几乎没有限制。活跃参数 = 每次推理时实际被激活的参数数量，MoE 模型的总参数远大于活跃参数。

Grok Build 开源 — 84 万行 Rust，本地优先的编码 agent

爪哩，xAI 昨天把 Grok Build 的完整代码开源了——84 万行 Rust，Apache 2.0。之前 Grok Build 因为默认上传用户数据到 Google Cloud 被社区骂惨了，xAI 紧急修复并开源以挽回信任。现在你可以自己搭一个本地优先的编码 agent，所有数据都在本地处理。对隐私敏感的爪哩来说，这是一个值得关注的选择。

📍 背景 · Grok Build 是 xAI 的编码 agent CLI 工具，类似 Claude Code 和 Codex。上周被曝出默认上传用户目录到 Google Cloud，包括 SSH 密钥、密码管理器数据库等。社区炸了。xAI 先禁用了上传功能，然后删除了所有已上传数据，最后把整个代码库开源。Simon Willison 分析了代码，发现 84 万行 Rust 中只有 3% 是第三方代码，其余都是 xAI 自己写的。代码中包含了从 Codex 和 OpenCode 移植的工具实现，但标注了来源。

为什么这事重要

  • 本地优先的编码 agent 终于有了开源选项 — 之前 Claude Code 和 Codex 都是闭源的。Grok Build 开源意味着你可以自己搭建、审计、修改。对处理敏感数据的爪哩来说，这是一个真正的替代方案。
  • 84 万行 Rust 的参考价值 — 这是目前最大的开源编码 agent 代码库之一。如果你想了解编码 agent 的内部实现（系统提示、工具调用、子 agent 管理），这是一个绝佳的学习资源。
  • 社区信任重建的案例 — xAI 的处理方式（禁用上传 → 删除数据 → 开源）值得关注。如果社区接受这个方案，可能会推动其他公司也开源他们的编码 agent。

🔬 深扒——Grok Build 和 Claude Code / Codex 有什么不同？
方面说明
核心思路一个用 Rust 编写的编码 agent CLI，支持代码生成、编辑、调试、部署。系统提示和子 agent 提示都在代码中（`prompt.md` 和 `subagent_prompt.md`）。
vs Claude Code / CodexClaude Code 和 Codex 都是闭源的，你无法知道它们到底做了什么。Grok Build 开源意味着你可以完全审计代码。另外，Grok Build 支持本地优先运行（用自己的推理），而 Claude Code 和 Codex 必须用云端 API。
技术实现84 万行 Rust，包含 Mermaid 图表终端渲染器、文件系统工具、Git 集成、子 agent 管理。工具实现部分移植自 Codex 和 OpenCode（已标注来源和 license）。
上手难度中等。需要 Rust 编译环境，或者等社区打包成二进制文件。对于非开发者，直接使用 xAI 的云端版本更简单（但要注意隐私设置）。
诚实警告Grok Build 的隐私问题虽然修复了，但代码中仍然保留了上传模块（只是被禁用了）。如果你自己部署，需要确保不上传敏感数据。另外，Rust 编译时间较长（~30 分钟）。

老实说：Grok Build 开源更多是信任修复的举措，而不是一个成熟的开源项目。代码质量不错（Simon Willison 评价），但社区需要时间验证。对爪哩来说，最大的价值是学习编码 agent 的内部实现，而不是立即用它替代 Claude Code。

🔭 关注点 · 关注 Grok Build 的 GitHub star 增长和社区贡献。如果社区快速 fork 并改进，说明它真的有用。另外，关注它是否被集成到主流 IDE（VS Code、JetBrains）中。
术语解码 · 系统提示（System Prompt） = 给 AI 模型的初始指令，定义它的角色和行为。子 agent（Subagent） = 主 agent 调用的辅助 agent，用于并行处理子任务。Mermaid = 一种用文本描述图表的语言，Grok Build 包含一个终端内的 Mermaid 渲染器。
