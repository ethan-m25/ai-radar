OpenCode — 开源编码 agent，token 消耗比 Claude Code 少 4 倍

爪哩，今天 HN 上有个帖子炸了——"Claude Code sends 33k tokens before reading the prompt; OpenCode sends 7k"。如果你每天用 Claude Code，这个对比直接关系到你的钱包。OpenCode 是一个开源的编码 agent，用更聪明的 prompt 管理和上下文压缩，把每次对话的 token 开销从 33k 降到了 7k。对于你这种高频用户，每月 token 账单可能减半。

📍 背景 · Claude Code 是 Anthropic 的官方编码 agent，集成在终端中，很多人（包括爪哩）天天用。但它的 token 消耗一直是个痛点——每次启动新对话，它会先发送大量系统 prompt 和上下文，导致 33k 的固定开销。OpenCode 是一个社区项目，声称通过优化 prompt 结构和缓存机制，将固定开销降到 7k。这个对比帖在 HN 上获得 604 points 和 327 条评论，说明社区对此非常关注。

为什么这事重要

  • 直接省钱 — 如果你每天用 Claude Code 处理 50 个对话，每个对话省 26k token，按 $10/M 输出 token 计算，每天省约 $13，一个月就是 $390。OpenCode 还开源，你可以自己部署，甚至用更便宜的 API。
  • 完全开源 — OpenCode 的代码在 GitHub 上，你可以审查、修改、自托管。不像 Claude Code 是封闭的，OpenCode 让你掌控自己的数据和工作流。
  • 对 vibe coding 的直接影响 — 你已经在用 Codex 和 Claude Code 做 office 自动化，OpenCode 可能成为更经济的选择，尤其适合大批量任务。而且它的开源生态意味着社区会持续改进。

🔬 深扒——OpenCode 和 Claude Code 到底有什么不同？
方面说明
核心思路OpenCode 通过更紧凑的系统 prompt、缓存上次对话的上下文、以及智能裁剪来减少每次请求的 token 开销。Claude Code 则采用更全面的系统 prompt，包含大量规则和示例。
vs Claude CodeClaude Code 官方、稳定、集成好，但 token 开销大且封闭。OpenCode 开源、便宜、可定制，但可能不够成熟，功能可能少一些（比如 MCP 支持、文件系统操作等）。
vs CodexCodex 是 OpenAI 的编码 agent，token 开销也较高。OpenCode 可能也支持多种后端（如 Claude、GPT-4），但帖子中主要对比的是 Claude Code。
技术实现具体技术细节未知，但通常开源 agent 会使用更轻量的 prompt 模板、上下文压缩算法（如 LLMLingua）或 KV cache 复用。
上手难度中等。需要安装 Python/Node.js，配置 API key。对于非开发者，可能需要一些命令行经验。但比你之前用 Claude Code 可能多一步安装。
诚实警告OpenCode 是年轻项目，可能不稳定，缺乏企业级支持。在关键生产力任务上，Claude Code 的可靠性可能更高。但作为实验和学习工具，非常值得尝试。

老实说：token 节省 4 倍听起来很诱人，但实际使用中可能因任务复杂度而异。OpenCode 的固定开销低，但后续对话的上下文管理可能不同。建议先在小项目上测试，对比结果。

🔭 关注点 · 如果 OpenCode 的 GitHub stars 在一周内超过 5000，说明社区认可度很高。另外，关注它是否添加了对 MCP 协议的支持（你之前用过的）。如果 Anthropic 因此降低 Claude Code 的 token 开销，那就更好了。
术语解码 · 固定 token 开销 = 每次开始新对话时，agent 必须发送的一批系统指令和上下文，这些 token 无论你问什么都要消耗，属于"固定成本"。KV cache = 一种缓存机制，可以复用之前计算过的注意力键值，减少重复计算。LLMLingua = 一种上下文压缩算法，能把 prompt 压缩到原来的一半甚至更少。
