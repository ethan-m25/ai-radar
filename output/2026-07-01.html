Claude Sonnet 5 — 性能接近 Opus 4.8，但 tokenizer 变化导致实际涨价 30%

clawii，Anthropic 今天发布了 Claude Sonnet 5，定位是"接近 Opus 4.8 的性能，更低的价格"。但 Simon Willison 发现它的新 tokenizer 让同样输入多出约 30% 的 token，相当于实际涨价 30%。这对你用 Claude API 做 vibe coding 的成本有直接影响。同时，Sonnet 5 有 1M token 上下文窗口和 128K 最大输出，支持 adaptive thinking，但不再支持 temperature/top_p/top_k 参数。

📍 背景 · Claude Sonnet 系列一直是 Anthropic 的"性价比"产品线。Sonnet 4.6 定价 $3/$15 per million tokens。Sonnet 5 官方定价相同，但使用了新的 tokenizer，导致英文文本 token 数增加约 30%，中文不变，代码增加约 28%。这意味着实际成本上升了约 30%。性能方面，Anthropic 称其接近 Opus 4.8（之前的最强模型），但低于 Mythos 5。Sonnet 5 的发布没有受到出口管制影响，因为其网络能力低于 Mythos 5。Simon Willison 用 Claude Token Counter 工具验证了 tokenizer 变化。

为什么这事重要

  • 实际成本上涨 — 虽然官方价格没变，但 tokenizer 变化导致英文和代码输入多消耗 30% token，相当于涨价 30%。如果你用 Claude API 做大量英文/代码任务，成本会明显上升。
  • 性能接近旗舰 — Sonnet 5 的性能接近 Opus 4.8，但价格只有 Opus 4.8 的几分之一（Opus 4.8 定价 $15/$75）。对于不需要 Mythos 级别能力的任务，Sonnet 5 是新的性价比选择。
  • 参数变化 — 不再支持 temperature/top_p/top_k，adaptive thinking 默认开启。这意味着你无法像以前那样精细控制生成随机性。对于需要确定性的任务（如代码生成），可能需要显式禁用 thinking。
  • 对你 vibe coding 的影响 — 如果你在 Claude Code 或 Codex 中使用 Sonnet 5，注意成本变化。中文用户受影响较小（token 数几乎不变），但英文 prompt 和代码会多花钱。
🎯 你现在能做的事：
  1. 测试你的实际成本 — 用 Simon Willison 的 Claude Token Counter 输入你常用的 prompt 和代码，看看 token 数增加了多少。
  2. 调整预算 — 如果使用 Sonnet 5，将 API 成本预算上调 30%。或者考虑在中文任务上继续使用 Sonnet 4.6（如果仍可用）。
  3. 禁用 adaptive thinking — 在 API 调用中设置 thinking: {type: "disabled"} 以获得更确定性的输出，同时可能减少 token 消耗。
  4. 对比 Opus 4.8 — 在关键任务上对比 Sonnet 5 和 Opus 4.8 的输出质量，看是否真的可以降级使用。
🔬 Deep dive — Claude Sonnet 5 的技术细节
  • vs Sonnet 4.6：性能大幅提升，接近 Opus 4.8。新 tokenizer 导致英文 token 增加 30%，代码增加 28%，中文几乎不变。定价相同但实际成本上升。
  • vs Opus 4.8：Sonnet 5 性能接近 Opus 4.8，但价格低得多（$3/$15 vs $15/$75）。Opus 4.8 仍可用于需要最高质量的任务。
  • 上下文窗口：1M token 输入，128K token 输出。与 Sonnet 4.6 相同。
  • 参数变化：temperature、top_p、top_k 不再支持。adaptive thinking 默认开启，除非显式禁用。这意味着模型会自动决定思考深度。
  • 安全限制：Sonnet 5 的网络能力低于 Mythos 5，因此未受出口管制影响。其安全措施与 Opus 4.7/4.8 类似。
  • 定价：$3/百万输入 token，$15/百万输出 token。2026年8月31日前有 introductory discount：$2/$10。
🔭 下一步触发条件：
• 如果 Claude Code 默认切换到 Sonnet 5，信号升级（意味着 Anthropic 认为它足够好）。
• 如果出现 Sonnet 5 vs Opus 4.8 的独立基准对比，值得关注。
• 如果社区发现 tokenizer 变化导致特定任务（如长文档摘要）成本暴增，可能引发 backlash。
术语解码：
Tokenizer = 将文本切分成 token 的工具。不同模型使用不同的 tokenizer，导致同一段文本的 token 数不同。Sonnet 5 的新 tokenizer 对英文和代码更"细粒度"，所以 token 数更多。
Adaptive thinking = 模型自动决定是否进行深度推理（thinking）。默认开启，但可以通过 API 参数禁用。

Claude Code 被发现在请求中嵌入隐写标记

clawii，今天 HN 头条第二名（2099 分，606 评论）是一篇博客，标题是"Claude Code is steganographically marking requests"。作者发现 Claude Code 在发送给 Anthropic API 的请求中嵌入了隐写标记（steganographic markers），这些标记对用户不可见，但可以被 Anthropic 用来追踪请求来源。这引发了关于隐私和透明度的激烈讨论。如果你在用 Claude Code，你的每一次代码补全请求都可能被标记。

📍 背景 · 隐写术（steganography）是将信息隐藏在看似正常的数据中。博客作者 kirushik 发现 Claude Code 在 HTTP 请求的某些字段中嵌入了独特的标记，这些标记可能用于识别用户、会话或请求来源。Anthropic 尚未公开回应。社区讨论集中在：这是否是正常的遥测技术？是否侵犯隐私？Claude Code 是闭源产品，用户无法验证其行为。类似的做法在 AI 编码助手领域并不罕见（如 GitHub Copilot 也有遥测），但隐写标记的方式更隐蔽。

为什么这事重要

  • 隐私问题 — 如果你用 Claude Code 处理公司代码或敏感信息，这些隐写标记可能被用来关联你的身份和代码内容。即使 Anthropic 声称不存储代码，标记本身可能泄露元数据。
  • 透明度缺失 — 隐写标记未被文档记录，用户不知情。这违背了 AI 工具透明度的最佳实践。
  • 对你 vibe coding 的影响 — 如果你在公司环境中使用 Claude Code，可能需要评估隐私风险。考虑使用本地模型（如 Qwen 3.6 27B）替代，或者等待 Anthropic 的官方说明。
🎯 你现在能做的事：
  1. 检查你的 Claude Code 请求 — 使用网络抓包工具（如 Wireshark 或 Charles Proxy）查看 Claude Code 发出的请求，看是否有异常字段。
  2. 评估隐私风险 — 如果你在公司处理敏感代码，暂时切换到本地模型（如 Qwen 3.6 27B 或 Ornith-1.0）或使用其他编码助手。
  3. 关注 Anthropic 回应 — 等待 Anthropic 官方说明。如果标记是用于防止滥用或调试，可能可以接受；如果是用于追踪用户行为，则问题更大。
  4. 考虑替代方案 — 如果隐私对你很重要，考虑使用完全离线的编码助手，如 Continue + 本地模型，或 Codex 的本地模式（如果支持）。
🔬 Deep dive — 隐写标记的技术细节
  • 发现过程：博客作者 kirushik 通过抓包分析 Claude Code 的网络请求，发现某些字段包含看似随机但实际有规律的数据，怀疑是隐写标记。
  • 可能用途：请求追踪、用户识别、会话关联、反滥用检测。类似 Google Analytics 的 cookie，但更隐蔽。
  • vs Copilot 遥测：GitHub Copilot 也会发送遥测数据，但通常明确告知用户。Claude Code 的隐写标记未被文档记录，这是主要争议点。
  • 影响范围：所有使用 Claude Code 的用户（包括 Claude Code CLI 和 IDE 插件）。Anthropic 的 API 调用也可能受影响。
  • 局限性：目前只有单一来源的发现，需要更多独立验证。Anthropic 可能很快发布声明。
🔭 下一步触发条件：
• 如果 Anthropic 官方确认并解释隐写标记的用途，信号可能降级或升级。
• 如果出现更多独立验证或发现其他 AI 工具也有类似行为，可能成为行业事件。
• 如果用户开始大规模迁移到其他编码助手，信号升级。
术语解码：
隐写术（Steganography） = 将秘密信息隐藏在普通数据中，使第三方不易察觉。这里指 Claude Code 在请求中嵌入不可见的标记。
遥测（Telemetry） = 自动收集使用数据并发送给服务提供商，用于改进产品或检测问题。

Claude Science — Anthropic 推出面向科学家的 AI 工作台

clawii，Anthropic 今天还发布了 Claude Science，一个可定制的 AI 工作台，集成了科学家常用的工具和包，能生成可审计的工件，并提供灵活的计算资源访问。HN 上 503 分，147 评论。虽然你大概率不是科学家，但这个产品展示了 Anthropic 向垂直领域扩展的思路——未来可能会有"Claude Finance"或"Claude HR"之类的东西。

📍 背景 · Claude Science 是 Anthropic 推出的第一个垂直领域产品。它不是一个通用聊天界面，而是一个专门为科学研究设计的工作台，集成了 Python、Jupyter、LaTeX、文献搜索等工具。它还能生成可审计的工件（如数据表格、图表、代码），方便科学家记录和复现实验。计算资源方面，它提供灵活的后端（可能是云端 GPU/CPU）。定价尚未公布，但预计会包含在 Claude Enterprise 或单独订阅中。

为什么这事值得看

  • 垂直领域产品化 — 这是 Anthropic 从通用模型向行业解决方案迈出的第一步。如果成功，后续可能会有更多垂直产品。
  • 可审计性设计 — Claude Science 强调生成可审计的工件，这对任何需要记录工作流程的行业都有参考价值。
  • 对你可能的意义 — 虽然你现在用不上，但 Anthropic 的产品思路值得关注。未来如果推出面向办公自动化的类似产品，你可能是目标用户。
🎯 你现在能做的事：
  1. 阅读官方公告 — 了解 Claude Science 的功能和设计理念，看看是否有可以借鉴到办公场景的思路。
  2. 关注定价 — 如果定价合理且功能扩展到其他领域，可能值得尝试。
  3. 保持观望 — 目前只面向科学家，非目标用户暂时不需要深入。
🔬 Deep dive — Claude Science 的功能
  • 集成工具：Python 环境、Jupyter Notebook、LaTeX 编辑器、文献搜索（可能集成 arXiv/PubMed）、数据可视化库。
  • 可审计工件：所有生成的内容（代码、图表、分析结果）都有版本记录，方便复现和审查。
  • 计算资源：提供云端 GPU/CPU，可能按使用量计费。与 Claude API 分开。
  • vs 通用 Claude：Claude Science 是专门为科学工作流优化的，不是简单的聊天界面。它更像一个"AI 增强的 Jupyter Lab"。
  • 局限性：目前只面向科学研究，功能可能过于专业。定价未知，可能较贵。
🔭 下一步触发条件：
• 如果 Anthropic 推出面向其他垂直领域（如金融、法律、HR）的类似产品，说明产品化战略加速。
• 如果 Claude Science 获得大量科学家用户，证明垂直产品模式可行。
术语解码：
可审计工件（Auditable artifacts） = 记录工作过程和结果的文件，可以被其他人检查和复现。在科学研究中，这相当于实验记录本。
